Self-hosting is a first-class part of the product story, not an afterthought.

## Baseline shape

The formal self-hosted baseline is:

- one `opentoggl` runtime image
- one `postgres` dependency
- one `redis` dependency

Schema management is handled through `pgschema`, and the compose workflow is documented in the repository itself. In the self-hosted image, the runtime entrypoint applies desired state from `DATABASE_URL` before the API process starts.

## Canonical startup order

1. Start `postgres`, `redis`, and `opentoggl`
2. The `opentoggl` entrypoint runs `pgschema apply`
3. Verify `/healthz`, `/readyz`, and an HTTP smoke check

That order is important because self-hosted startup is defined around the same schema truth and readiness semantics as the rest of the repository.

## What operators control

- deployment timing
- infrastructure placement
- environment overrides
- upgrade and rollback cadence
- persistent PostgreSQL data

## What stays the same

- the public product contract
- the self-hosted runtime shape documented in the repo
- the readiness and smoke verification path

## Smoke verification

Minimum smoke verification is intentionally small and explicit:

```bash
curl -fsS http://localhost:8080/healthz
curl -fsS http://localhost:8080/readyz
curl -fsSI http://localhost:8080/
```

If these checks fail, the self-hosted baseline is not considered ready.
