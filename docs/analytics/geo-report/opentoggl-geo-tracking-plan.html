Module 01

企业理解摘要

#01

OpenToggl 是开源、可自托管的 Toggl Track 替代品，官方站点分为 landing（opentoggl.com，SSR，多语言）与 demo（track.opentoggl.com，SPA，noindex），核心代码仓库在 github.com/CorrectRoadH/opentoggl。

从 GEO 后端归因角度看，这类公司需要把归因目标拆成四层：GitHub Star / Fork、demo 试用、自托管安装、文档深读；只看 landing pageview 会漏掉真正定义产品价值的安装与留存行为。

  • 已验证事实：apps/landing 根布局 root.tsx 已加载 Ahrefs Web Analytics 脚本。
  • 已验证事实：apps/landing/app/lib/utm.ts 提供 appendUtm，home.tsx 已给 GitHub / demo 外链挂上 utm_source=opentoggl_landing 等参数。
  • 已验证事实：apps/website/index.html 为 noindex, nofollow 的 SPA，且未加载任何 web analytics。
  • 已验证事实：apps/update-worker 使用 Cloudflare Analytics Engine 记录自托管实例版本、OS、架构、country（blobs 1-6）。
  • 推断：由于 GitHub 不读外链 UTM，现状下 hero_cta / proof_card / footer 等位置级点击归因无法在任何后台恢复。
  • 推断：landing 到 demo 的转化路径当前断裂，所有来自 AI 平台的线索在 demo 侧会被计为 direct 或 unknown。

Module 02

市场分层判断

#02

OpenToggl 当前主要面向海外开源开发者社区，landing 默认语言为英文，核心发现场景是 GitHub、Hacker News、Reddit、X、以及 ChatGPT / Gemini / Perplexity / Claude 给出的 open-source time tracker 推荐。

因此方案主骨架放在海外 GEO：用官网页矩阵、表单字段、页面事件、自托管安装回传作为主链路，不为国内 GEO 配专属企微 / 电话 / 口令等承接资产。国内 GEO 仅保留 Brand Radar 监控与中文 landing 曝光观察，等有明确信号再补。

  • 海外 GEO：ChatGPT / Gemini / Perplexity / Claude 为核心；主归因依赖 landing/demo 页矩阵、/go/:target 中转、install_source 回传。
  • 国内 GEO：DeepSeek / 豆包 / 元宝 / Kimi 仅做 Brand Radar 监控；暂不建立口令、企微、电话等来源补丁。
  • 混合收口：所有归因事件写入同一张 inbound_event 表，ai_platform 字段按 enum 存，便于后续按国内 / 海外拆分。

现状可以拆成已具备资产与明显缺口两类，直接决定后续优先级路线图。

  • 已具备：Ahrefs Web Analytics（landing pageview / referrer / country）。
  • 已具备：Ahrefs 订阅覆盖 Brand Radar / GSC / Keywords Explorer / Site Explorer。

  • 缺口：GitHub 外链无位置级归因（UTM 对 GitHub 不生效，且 GitHub Traffic Insights 仅留 14 天）。

  • 缺口：Google Search Console / Bing Webmaster 无周度报表机制。
  • 缺口：GEO prompt 监控 / Brand Radar 尚未启用，AI 答案曝光零可观测。
  • 缺口：自托管 docker run / 二进制安装的 install_source 字段缺失。

#06

AI 曝光 → 承接 → 识别 → 归因 → 复盘 链路

阶段 1

AI 曝光

固定 prompt 在 ChatGPT / Gemini / Perplexity / Claude 查询，Brand Radar 记录 mention 与 cited-pages。

阶段 2

站外承接

用户点击 AI 答案里的链接，或自行搜 opentoggl 抵达 opentoggl.com / GitHub。

阶段 3

识别

阶段 5

复盘

Module 07

直接效果追踪方案

#07

直接效果的设计目标是让每一次点击 / 试用 / 安装都尽量带上一个可被写回来源主键的 slot 标签；优先覆盖最容易丢失归因的 GitHub 外链与自托管安装两条路径。

  • D1 /go/:target 中转：Cloudflare Worker 路由 opentoggl.com/go/:target?ref=<slot>&locale=<xx>；白名单 github / github-issues / github-discussions / demo / x / status / hn；fail-open 直接 302；写 Analytics Engine (index1=target, blob1=slot, blob2=locale, blob3=referer, blob4=country, blob5=ua_family, blob6=visitor_id)。
  • D2 跨域 visitor stitching：landing 写 1st-party cookie ot_v（Domain=.opentoggl.com, SameSite=Lax），/go/demo 把 ot_v 透传到 track.opentoggl.com?ot_v=...&src=...；demo bootstrap 读 ot_v/src → 发 first_visit 到 /api/ingress/first-visit → 从 URL 清理参数。
  • D3 install_source 回传：update-worker 新增 install_source / install_campaign query；curl install.sh 默认带 INSTALL_SOURCE=github_readme；docker-compose 模板写 x-install-source=docs_docker_compose。
  • D4 demo 首问问卷：demo 首次 onboarding 单选 How did you hear about OpenToggl，选项覆盖 ChatGPT / Claude / Perplexity / Gemini / GitHub / Google / X / HN / Reddit / Friend / Other；写入 user_attribution 表，一用户一条。
  • Slot 命名规范：hero_cta / proof_card / footer / faq / docs_inline / readme_cta；仓库内以 config/outbound-targets.json 收口，PR review。

Module 08

间接效果追踪方案

#08

间接效果用来覆盖直接归因不可达的场景：AI 私有会话、口碑扩散、用户从 AI 看到后再自己 Google 搜、GitHub 仓库被别的 README 提及等。这一层的核心是趋势判断，不是逐条归因。

  • I1 SEO：验证 opentoggl.com 与 track.opentoggl.com 两个 GSC property（DNS TXT）；通过 Ahrefs MCP gsc-keywords / gsc-pages / gsc-performance-history 每周自动拉报表。
  • 周度 SEO 四图：品牌词曝光 / 点击 / CTR；非品牌 query Top 20 涨跌（按 locale）；高曝光低 CTR 清单（title / description 优化线索）；page-level 曝光 Top 20。
  • I2 Brand-level 信号：每天抓 repos/{owner}/{repo} API，留 github_metrics_daily.jsonl；周对比 Brand Radar mention、GitHub star、GSC 品牌词三条趋势。
  • SoV 监控：Brand Radar 监控 OpenToggl 与竞品 Toggl Track / Clockify / Harvest / Timely / TimeCamp 的 share of voice。
  • 不做：Google Analytics、cookie-based 品牌指纹识别、用户级长期画像。

Module 09

监测效果边界说明

#09

GEO 的完整价值通常不能被任何单一后台直接统计，OpenToggl 尤其如此。下列链路属于结构性不可观测，不应承诺能用后台数据补全。

因此 20%~30% 的保守规划假设对 OpenToggl 不直接适用。OpenToggl 官网承接较强但用户后续 self-host 安装链路非常松散，应使用 海外官网承接强、表单成熟的 SaaS / PLG 区间作为起点，并对 Unobservable 层相对加权。

  • ChatGPT Pro / Claude Pro 私有会话里被引用到 OpenToggl 的对话，后台永远看不到。
  • 用户在 AI 答案里看到 OpenToggl 后不点链接，第二天自行 Google 搜 opentoggl 进入，这部分会被误计为 organic search 而非 GEO。
  • AI 推荐后的口头推荐、Slack / Discord 私聊推荐，没有任何数字痕迹。
  • self-host 的价值实现发生在安装与运行后，但安装脚本的来源完全依赖用户是否带 INSTALL_SOURCE，覆盖率天生不全。
  • 多数 AI 平台不在答案里稳定输出引用链接（Perplexity 稳定、ChatGPT 不稳定、Claude 以引用内容为主不给链接），referer 这一条在 GEO 场景天然稀缺。

  • outbound_click / demo pageview → Cloudflare Analytics Engine。

Module 13

置信度与缺口

#13
  • 高置信度：/go/:target 中转、GSC 接入、Brand Radar 接入，这三条是成熟路径。
  • 中置信度：跨域 visitor stitching；Safari ITP / Firefox ETP 下 1st-party cookie TTL 可能被缩到 7 天，对长周期归因有损。
  • 低置信度：install_source 覆盖率，取决于用户是否改 install 命令；不强制、不 block release。
  • 缺口 1：国内 GEO 的实际流量占比未知；等 Brand Radar 数据回来再决定是否补 DeepSeek / 豆包 / 元宝 / Kimi 专项监控。
  • 缺口 2：self-hosted 实例活跃度与来源的相关性未知，update-worker 需要多跑几周积累样本。
  • 缺口 3：GEO prompt 库初始版需 1-2 次迭代才稳定；v1 只是起点不是答案。
