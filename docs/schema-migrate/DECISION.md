## 目的

这份文档不重复定义目标，也不重写各方案实现细节。

这里只回答 4 个问题：

- 为什么 `PLAN-1` / `PLAN-2` / `PLAN-3` / `PLAN-4` 现在都保留
- 当前更推荐哪一个作为主路径
- 为什么当前不把其它方案作为主路径
- 在什么前提下，推荐结论可能改变

---

## 先说明前提

- 当前仓库正式架构仍以 `pgschema` 为唯一正式 schema 管理路径
- 因此，`PLAN-1`、`PLAN-2`、`PLAN-4` 都不是普通实现细节调整，而是在挑战当前正式架构结论
- `PLAN-3` 则是不改变当前正式架构前提下的保守方案

这意味着：

- 保留多个 plan 是合理的，因为它们代表的是不同层级的决策
- 当前推荐哪个 plan，取决于团队是在做"终局架构选择"，还是在做"短期过渡决策"

---

## 为什么保留四个 Plan

### 保留 `PLAN-1`

- 它代表"以显式版本化迁移为主路径"的长期方案
- 它直接回应复杂迁移、可审阅过程、self-hosting 可重复执行这几类核心目标
- 它是当前候选方案里最成熟、最常见的 Go 生态路径之一

### 保留 `PLAN-2`

- 它代表"同时强化声明式 schema 治理和版本化迁移"的长期方案
- 它不是当前最短路径，但对长期 drift detection、lint 和 schema 治理有额外价值
- 保留它有助于避免把"现在不选"误读成"以后永远不选"

### 保留 `PLAN-3`

- 它代表"先不推翻当前架构定论"的过渡方案
- 但需要明确：它不是"暂时够用"，而是**所有复杂变更都退化成在正式体系之外人工操作数据库、再让 pgschema 追认**
- 保留它有助于把"短期不切换"说清楚，而不是假装没有这个选项

### 保留 `PLAN-4`

- 它代表"保留版本化迁移主路径，但尽量贴近 pgx/native SQL 风格"的长期方案
- 它回应的是另一个真实偏好：不一定要引入 `goose` 这样的更重框架，也可以选择更贴近现有 Go/pgx 栈的路线
- 保留它有助于把"要不要 Go migration 框架能力"这个问题和"要不要转向 versioned migrations"拆开讨论

## 当前推荐

如果目标是现在选一条长期主路径，当前更推荐 [PLAN-1.md](PLAN-1.md)。

### 推荐理由

- 它能把复杂迁移、显式迁移过程、执行历史和 self-hosting 执行路径纳入正式机制
- 它比 `PLAN-2` 更轻，不会在第一步同时引入声明式 schema 治理和 migration 生成两层额外复杂度
- 它满足单二进制交付这个明确功能需求，不需要操作者额外拿到独立 schema CLI
- **goose 是四个方案里长期维护成本最低的版本化迁移路径**

### 为什么选 goose 而不是 tern

`PLAN-1`（goose）和 `PLAN-4`（tern）解决的问题完全相同。两者的核心差异不在能力，而在工程选型的务实考量：

**1. "贴近 pgx 风格"的收益被高估了**

tern 的主要卖点是直接接受 `*pgx.Conn`，不需要 `database/sql` 适配层。但 OpenToggl 用的是 `pgxpool.Pool`，tern 同样需要 `pool.Acquire()` 再取 `conn.Conn()`。goose 需要 `stdlib.OpenDBFromPool(pool)`。两边的集成差异是几行代码，不是架构级差异。

这是一次性写完就不会再碰的胶水代码，不应该成为选型的主因。

**2. 社区成熟度的差距是真实的长期成本**

| 维度 | goose | tern |
|------|-------|------|
| Stars | 10,400+ | 1,275 |
| Forks | 632 | 86 |
| 生产使用案例 | 大量，容易搜到 | 少，遇到问题主要靠读源码 |
| 第三方集成示例 | 丰富 | 稀少 |

对 1-2 人的团队，"遇到问题能 Google 到答案"是真实的效率差。这个差距在日常简单 migration 中感知不到，但在处理 baseline 策略、CI 校验、多分支冲突、生产故障排查时会暴露出来。

**3. goose 有 tern 缺失的防御性功能**

- **checksum 校验**：goose v3 会校验已执行 migration 文件是否被篡改。tern 没有这个能力。如果有人改了已执行的 migration 文件，tern 不会报错，goose 会。
- **out-of-order migration**：goose 支持 `--allow-missing`，允许补执行漏掉的 migration。tern 严格顺序，多分支并行开发时容易冲突。

这两个功能在早期不一定用到，但一旦需要时，自己补齐的成本远高于框架自带。

**4. tern 的优势不构成决定性理由**

- **内置 advisory lock**：goose v3 也支持。不是差异点。
- **Go migration 拿 `*pgx.Conn`**：方便，但也意味着更容易把 migration 代码写成依赖当前项目实现习惯的代码，增加历史 migration 腐化风险。goose 的 `*sql.DB` 反而形成一层自然隔离。
- **模板语法**：SQL migration 里写 Go template 是双刃剑，容易降低可读性。OpenToggl 不需要这个能力。
- **依赖更轻**：goose 的依赖也不重，这不是实际瓶颈。

### 总结

tern 不是错的选择。但 goose 在社区成熟度、防御性功能、多分支协作支持上都更强，而 tern 的优势（pgx 原生集成、内置 advisory lock）在实际集成中只节省了几行胶水代码。

**选型应该优先考虑长期维护成本，而不是初始集成的风格偏好。**

## 为什么当前不选 `PLAN-4` 作为主路径

- `PLAN-4` 可行，tern 本身维护状态健康（v2.3.6，2026-03-28 发布，jackc 持续维护）
- 但"贴近 pgx 风格"是偏好判断，不是客观优势，不应该成为选型主因
- 社区规模差 8 倍（10,400 vs 1,275），在边缘场景排查、最佳实践参考上是真实的效率差
- 缺少 checksum 校验和 out-of-order migration 支持，团队需要自己补齐或接受风险
- runner 边界、baseline 策略、`latest.sql` 漂移检测、Go migration 纪律约束，goose 同样需要团队自己补，但 goose 生态里能找到更多现成参考

什么时候会改选 `PLAN-4`：

- 团队明确认为 pgx 原生集成的开发体验优先于社区成熟度
- 团队接受 tern 的功能边界，不需要 checksum 校验和 out-of-order migration
- 团队愿意在更小的社区中自行解决边缘问题

---

## 为什么当前不选 `PLAN-2` 作为主路径

- `PLAN-2` 同时引入声明式 schema 管理、migration 生成、运行时执行三层变化
- 当前最迫切的问题是"迁移过程要显式可控"，不是"缺一个更强的 diff 生成器"
- 在复杂迁移场景里，`Atlas` 最终仍然需要人工设计迁移过程
- 因此它更像第二阶段增强，而不是第一步主路径

什么时候会改选 `PLAN-2`：

- 团队中长期想把 drift detection、lint、schema diff 自动化也一并建设
- 团队愿意承担更高的工具引入成本
- 当前目标不只是替换 `pgschema`，还包括建设更强的 schema 治理体系

---

## 为什么当前不选 `PLAN-3` 作为长期主路径

- `PLAN-3` 可以保留当前 desired-state SQL 的阅读优势和短期最小改动成本
- 但它没有真正解决复杂迁移如何进入正式、可重复、可验证路径的问题
- 所有复杂变更都退化成"先手动操作数据库，再更新 schema.sql 让 pgschema 追认"——这意味着生产迁移过程不可复现、self-hosted 用户无法自动完成升级、CI 无法验证迁移路径
- 这不是"暂时够用"，而是在正式体系之外运维

所以：

- `PLAN-3` 只适合作为过渡方案
- 不适合作为终局方案

什么时候会短期选 `PLAN-3`：

- 当前阶段必须先稳住其它大改动
- 团队暂时不能承担迁移体系切换成本
- 团队明确接受它只是阶段性过渡，不把它误当成终局

---

## 维持现状的风险

如果既不选任何 plan、也不做 PLAN-3 的过渡约束，继续按当前方式运作：

- [GOALS.md](GOALS.md) 中记录的痛点会持续发生：FK 变更不可控、表重建无法区分意图、字段类型变更在有数据时直接报错
- 每次复杂 schema 变更都依赖人工在 pgschema 体系之外操作，没有版本化记录
- self-hosted 用户升级需要手工编排额外步骤，与单二进制交付的目标冲突
- 测试路径（直接执行 schema.sql）和生产路径（pgschema apply）继续分离

这些不是假设场景，而是最近 commit 历史中已经发生的问题。

## 如何选择

### 选 `PLAN-1`，如果当前目标是：

- 给 OpenToggl 确立长期正式的迁移主路径
- 优先解决复杂迁移与可审阅迁移过程
- 满足单二进制交付
- 希望正式版启动时自动判断并执行 migration
- 优先考虑长期维护成本和社区支撑

### 选 `PLAN-4`，如果当前目标是：

- 同样要转向长期 migration 主路径
- 明确偏好 pgx 原生集成的开发体验
- 接受 tern 的社区规模和功能边界

### 选 `PLAN-2`，如果当前目标是：

- 除了迁移执行，还要同时建设更强的 schema 治理体系
- 团队愿意接受更高的工具引入成本和工作流复杂度
- 目标不只是替换 `pgschema`，还包括强化 drift/lint/diff 自动化

### 选 `PLAN-3`，如果当前目标是：

- 短期先稳住其它大改动
- 当前阶段暂时不能承担迁移体系切换成本
- 接受它只是过渡，不把它误当成终局

---

## 当前结论

- 保留四个 plan 是合理的，因为它们分别覆盖长期主路径、增强型长期路径、短期过渡路径，以及更贴近现有栈的长期主路径
- 当前如果要给出主推荐，优先推荐 [PLAN-1.md](PLAN-1.md)——goose 在社区成熟度、防御性功能、多分支协作支持上的优势，比 tern 的 pgx 原生集成节省的几行胶水代码更有长期价值
- 当前如果明确偏好 pgx 原生集成优先于社区成熟度，则 [PLAN-4.md](PLAN-4.md) 是明确次选
- 当前如果只允许做最小改动、但不做终局承诺，则可暂时采用 [PLAN-3.md](PLAN-3.md)
- [PLAN-2.md](PLAN-2.md) 更适合作为后续增强路线，而不是第一步主路径
