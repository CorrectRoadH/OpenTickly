## 目的

这份文档不重复定义目标，也不重写各方案实现细节。

这里只回答 4 个问题：

- 为什么 `PLAN-1` / `PLAN-2` / `PLAN-3` / `PLAN-4` 现在都保留
- 当前更推荐哪一个作为主路径
- 为什么当前不把其它方案作为主路径
- 在什么前提下，推荐结论可能改变

---

## 先说明前提

- 当前仓库正式架构仍以 `pgschema` 为唯一正式 schema 管理路径
- 因此，`PLAN-1`、`PLAN-2`、`PLAN-4` 都不是普通实现细节调整，而是在挑战当前正式架构结论
- `PLAN-3` 则是不改变当前正式架构前提下的保守方案

这意味着：

- 保留多个 plan 是合理的，因为它们代表的是不同层级的决策
- 当前推荐哪个 plan，取决于团队是在做“终局架构选择”，还是在做“短期过渡决策”

---

## 为什么保留四个 Plan

### 保留 `PLAN-1`

- 它代表“以显式版本化迁移为主路径”的长期方案
- 它直接回应复杂迁移、可审阅过程、self-hosting 可重复执行这几类核心目标
- 它是当前候选方案里最成熟、最常见的 Go 生态路径之一

### 保留 `PLAN-2`

- 它代表“同时强化声明式 schema 治理和版本化迁移”的长期方案
- 它不是当前最短路径，但对长期 drift detection、lint 和 schema 治理有额外价值
- 保留它有助于避免把“现在不选”误读成“以后永远不选”

### 保留 `PLAN-3`

- 它代表“先不推翻当前架构定论”的过渡方案
- 如果当前阶段不能承担 migration 体系切换成本，它是现实的保守路径
- 保留它有助于把“短期不切换”说清楚，而不是假装没有这个选项

### 保留 `PLAN-4`

- 它代表“保留版本化迁移主路径，但尽量贴近 pgx/native SQL 风格”的长期方案
- 它回应的是另一个真实偏好：不一定要引入 `goose` 这样的更重框架，也可以选择更贴近现有 Go/pgx 栈的路线
- 保留它有助于把“要不要 Go migration 框架能力”这个问题和“要不要转向 versioned migrations”拆开讨论

---

## 当前推荐

如果目标是现在选一条长期主路径，当前更推荐 [PLAN-4.md](PLAN-4.md)。

推荐理由：

- 它和 `PLAN-1` 一样，能够把复杂迁移、显式迁移过程、执行历史和 self-hosting 执行路径纳入正式机制
- 它比 `PLAN-2` 更轻，不会在第一步同时引入声明式 schema 治理和 migration 生成两层额外复杂度
- 它比 `PLAN-1` 更贴近 OpenToggl 当前 Go + pgx 的实现风格，运行时边界更容易保持简单
- 它满足单二进制交付这个明确功能需求，不需要操作者额外拿到独立 schema CLI

这个推荐不是说 `PLAN-4` 没有成本，而是说：

- 在“要切到版本化迁移主路径”的前提下
- 它目前是四个方案里更平衡的一条路线

---

## 为什么当前不选 `PLAN-1` 作为主路径

- `PLAN-1` 可行，而且长期方向是对的
- 但它会引入 `goose` 的框架语义、Go migration 组织方式和额外约定
- 如果 OpenToggl 当前更重视“沿着现有 pgx/native SQL 风格演进”，`PLAN-4` 更贴近现有代码库习惯
- 因此当前不否定 `PLAN-1`，只是把它放在次选位置

什么时候会改选 `PLAN-1`：

- 团队明确希望复用成熟 Go migration 框架能力
- 团队接受 `goose` 的运行模型和约定
- `PLAN-4` 所需自定义 runner 设计被证明会补出太多框架能力

---

## 为什么当前不选 `PLAN-2` 作为主路径

- `PLAN-2` 同时引入声明式 schema 管理、migration 生成、运行时执行三层变化
- 当前最迫切的问题是“迁移过程要显式可控”，不是“缺一个更强的 diff 生成器”
- 在复杂迁移场景里，`Atlas` 最终仍然需要人工设计迁移过程
- 因此它更像第二阶段增强，而不是第一步主路径

什么时候会改选 `PLAN-2`：

- 团队中长期想把 drift detection、lint、schema diff 自动化也一并建设
- 团队愿意承担更高的工具引入成本
- 当前目标不只是替换 `pgschema`，还包括建设更强的 schema 治理体系

---

## 为什么当前不选 `PLAN-3` 作为长期主路径

- `PLAN-3` 可以保留当前 desired-state SQL 的阅读优势和短期最小改动成本
- 但它没有真正解决复杂迁移如何进入正式、可重复、可验证路径的问题
- 一旦遇到表重建、类型转换、分阶段删列、数据回填，流程仍然会退化成额外人工步骤
- 这和 self-hosting 升级、CI 验证、生产可复现之间存在长期冲突

所以：

- `PLAN-3` 适合作为过渡方案
- 但不适合作为终局方案

什么时候会短期选 `PLAN-3`：

- 当前阶段必须先稳住其它大改动
- 团队暂时不能承担迁移体系切换成本
- 团队明确接受它只是阶段性过渡，不把它误当成终局

---

## 如何选择

### 选 `PLAN-4`，如果当前目标是：

- 给 OpenToggl 确立长期正式的迁移主路径
- 优先解决复杂迁移与可审阅迁移过程
- 满足单二进制交付
- 尽量贴近现有 Go + pgx + SQL 的实现风格

### 选 `PLAN-1`，如果当前目标是：

- 同样要转向长期 migration 主路径
- 希望尽量复用成熟 Go migration 框架能力
- 接受 `goose` 带来的额外约定和运行模型

### 选 `PLAN-2`，如果当前目标是：

- 除了迁移执行，还要同时建设更强的 schema 治理体系
- 团队愿意接受更高的工具引入成本和工作流复杂度
- 目标不只是替换 `pgschema`，还包括强化 drift/lint/diff 自动化

### 选 `PLAN-3`，如果当前目标是：

- 短期先稳住其它大改动
- 当前阶段暂时不能承担迁移体系切换成本
- 接受它只是过渡，不把它误当成终局

---

## 当前结论

- 保留四个 plan 是合理的，因为它们分别覆盖长期主路径、增强型长期路径、短期过渡路径，以及更贴近现有栈的长期主路径
- 当前如果要给出主推荐，优先推荐 [PLAN-4.md](PLAN-4.md)
- 当前如果接受成熟框架优先于原生风格，则 [PLAN-1.md](PLAN-1.md) 是明确次选
- 当前如果只允许做最小改动、但不做终局承诺，则可暂时采用 [PLAN-3.md](PLAN-3.md)
- [PLAN-2.md](PLAN-2.md) 更适合作为后续增强路线，而不是第一步主路径
